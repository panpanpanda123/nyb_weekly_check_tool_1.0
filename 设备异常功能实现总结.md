# 设备异常监控功能实现总结

## 实现内容

### 1. 数据库模型 (shared/database_models.py)

新增两个数据表：

#### EquipmentStatus (设备异常状态表)
- 存储离线设备信息
- 关联门店ID、战区、区域经理
- 区分设备类型（POS/机顶盒）

#### EquipmentProcessing (设备异常处理记录表)
- 存储处理动作（已处理/不配合/特殊情况）
- 记录处理时间和理由

### 2. 数据导入脚本 (import_equipment_data.py)

功能：
- 读取3个Excel文件（收银设备、机顶盒、门店档案）
- 筛选离线设备
- 排除暂停营业门店
- 匹配whitelist获取战区和区域经理信息
- 导入到数据库

### 3. 后端API (viewer/app_viewer.py)

新增5个API端点：

1. `GET /equipment` - 设备异常监控页面
2. `GET /api/equipment/filters` - 获取筛选选项（战区、区域经理）
3. `GET /api/equipment/search` - 搜索设备异常（按门店分组，支持分页）
4. `POST /api/equipment/process` - 处理设备异常
5. `GET /api/equipment/export` - 导出处理结果

### 4. 前端页面

#### HTML (viewer/templates/equipment.html)
- 设备异常监控页面
- 选项卡切换（周清审核 | 门店评级 | 设备异常）
- 筛选区域（战区、区域经理、门店搜索）
- 统计信息和分页

#### CSS (viewer/static/equipment.css)
- 卡片式布局
- 响应式设计
- 处理状态徽章样式

#### JavaScript (viewer/static/equipment.js)
- 筛选和搜索逻辑
- 门店卡片渲染
- 处理动作交互
- 分页功能
- 导出功能

### 5. 选项卡集成

更新了3个页面的选项卡：
- viewer.html (周清审核)
- rating.html (门店评级)
- equipment.html (设备异常)

所有页面都可以通过选项卡快速切换。

### 6. 测试和部署

- 本地测试脚本: `本地测试设备异常.bat`
- 部署指南: `设备异常功能部署指南.md`

## 业务逻辑

### 数据处理流程

1. **数据收集**
   - 收银设备表 → 筛选"离线"状态
   - 机顶盒表 → 筛选"离线"状态
   - 门店档案表 → 获取暂停营业门店列表

2. **数据过滤**
   - 排除暂停营业门店
   - 匹配whitelist获取战区和区域经理
   - 只保留正常营业但设备离线的门店

3. **数据展示**
   - 按门店分组显示
   - 同一门店的多个设备合并在一张卡片中
   - 支持战区和区域经理筛选

4. **处理流程**
   - 区域经理查看自己负责的异常门店
   - 选择处理动作：已处理/不配合/特殊情况
   - 特殊情况需要填写理由
   - 处理记录永久保存

### 用户交互流程

1. 访问设备异常页面
2. 选择战区和区域经理进行筛选
3. 查看异常门店列表（每个卡片显示门店信息和设备列表）
4. 对每个门店进行处理：
   - 点击"已处理"：标记为已处理
   - 点击"不配合"：标记为不配合
   - 点击"特殊情况"：显示输入框，填写理由后提交
5. 导出处理结果Excel文件

## 技术特点

1. **按门店分组**: 同一门店的多个设备异常合并显示，避免重复
2. **分页加载**: 每页显示20个门店，提高性能
3. **实时筛选**: 支持战区、区域经理、门店搜索的组合筛选
4. **状态保存**: 处理记录永久保存在数据库中
5. **数据导出**: 支持导出所有设备异常和处理记录

## 注意事项

1. **数据更新**: 设备异常数据需要定期更新（建议每周一次）
2. **处理记录**: 处理记录不会被清空，会永久保存
3. **暂停营业**: 暂停营业门店会自动排除，不会显示
4. **设备合并**: 同一门店的多个设备会合并显示在一张卡片中
5. **权限控制**: 当前无登录验证，所有人都可以查看和处理
