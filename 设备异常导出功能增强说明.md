# 设备异常导出功能增强说明

## 新增功能

已增强设备异常导出功能，添加了区域经理的处理操作记录统计，方便查看每家店的处理情况和预计恢复期内的跳过状态。

## 新增导出列

### 1. 处理次数统计（3列）
- **近X天处理次数**：区域经理对该门店该设备的总处理次数
- **近X天已恢复次数**：标记"已恢复"的次数
- **近X天未恢复次数**：标记"未恢复"的次数

默认统计近10天，可通过URL参数调整：
```
/api/equipment/export?history_days=7   # 统计近7天
/api/equipment/export?history_days=14  # 统计近14天
```

### 2. 最近处理记录（1列）
显示最近5条处理记录的详细信息，格式：
```
03-11 10:30 已恢复；03-10 17:00 未恢复(设备故障)；03-09 11:00 已恢复
```

每条记录包含：
- 日期时间（月-日 时:分）
- 处理动作（已恢复/未恢复）
- 原因（如果有，显示前10个字符）

### 3. 预计恢复期内跳过（1列）
显示该门店是否在预计恢复期内被跳过，格式：
- **否**：正常显示，需要处理
- **是（预计03-15恢复）**：在预计恢复期内，暂时不提示

## 使用场景

### 场景1：查看区域经理处理频率
通过"近10天处理次数"列，可以看到：
- 哪些门店被频繁处理（可能是反复出问题）
- 哪些门店从未被处理（可能被忽略）
- 区域经理的工作量分布

### 场景2：分析处理效果
通过"已恢复次数"和"未恢复次数"对比：
- 已恢复次数多：说明问题能快速解决
- 未恢复次数多：说明问题难以解决，需要重点关注

### 场景3：查看处理历史
通过"最近处理记录"列：
- 查看处理时间分布（上午/下午）
- 查看处理动作变化（已恢复→未恢复→已恢复）
- 查看处理原因（如果填写了）

### 场景4：验证预计恢复期跳过
通过"预计恢复期内跳过"列：
- 确认标记"未恢复"且填写预计恢复日期的门店是否正确跳过
- 查看预计恢复日期是否合理
- 到期后是否恢复正常提示

## 技术实现

### 1. 查询近期处理记录
```python
# 获取近期所有处理记录
history_cutoff = datetime.now() - timedelta(days=history_days)
all_processing_records = session.query(EquipmentProcessing)\
    .filter(EquipmentProcessing.processed_at >= history_cutoff)\
    .order_by(EquipmentProcessing.processed_at.desc())\
    .all()
```

### 2. 统计处理次数
```python
# 按门店和设备类型分组
history_records = processing_history_dict.get(key, [])
total_processing_count = len(history_records)
recovered_count = sum(1 for r in history_records if r.action == '已恢复')
not_recovered_count = sum(1 for r in history_records if r.action == '未恢复')
```

### 3. 生成处理记录详情
```python
# 最近5条处理记录
processing_details = []
for record in history_records[:5]:
    detail = f"{record.processed_at.strftime('%m-%d %H:%M')} {record.action}"
    if record.reason:
        detail += f"({record.reason[:10]}...)"
    processing_details.append(detail)
```

### 4. 检查预计恢复期跳过
```python
# 使用现有的should_suppress函数
from equipment_utils import should_suppress
is_suppressed = should_suppress(session, store_id, equipment_type)
if is_suppressed:
    return f"是（预计{expected_recovery_date}恢复）"
return "否"
```

## 注意事项

1. **处理记录保留期**：默认保留10天的处理记录（配置：`PROCESSING_RETENTION_DAYS`）
2. **导出参数**：`history_days`不能超过`PROCESSING_RETENTION_DAYS`，否则数据不完整
3. **性能考虑**：导出大量数据时可能需要几秒钟，请耐心等待
4. **Excel列宽**：已自动调整列宽，"最近处理记录"列宽度为50，方便查看

## 相关配置

在 `equipment_config.py` 中：
```python
# 处理记录保留天数
PROCESSING_RETENTION_DAYS = 10
```

如果需要统计更长时间的处理记录，可以增加此配置值，但会占用更多数据库空间。

## 故障排查

### 问题1：导出失败
- 检查数据库连接
- 查看错误日志：`sudo journalctl -u review-viewer -n 50`
- 重启服务：`sudo systemctl restart review-viewer`

### 问题2：处理次数为0
- 检查是否有处理记录：`python 查看处理记录.py --days 10`
- 检查`history_days`参数是否正确
- 检查处理记录是否被清理（超过保留天数）

### 问题3：最近处理记录为空
- 检查该门店是否真的有处理记录
- 检查处理记录的时间是否在统计范围内
- 检查数据库中的`equipment_processing`表

### 问题4：预计恢复期跳过显示错误
- 检查`should_suppress`函数逻辑
- 检查`expected_recovery_date`和`suppressed_until`字段
- 检查当前日期是否在预计恢复期内
