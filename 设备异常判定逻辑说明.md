# 设备异常判定逻辑说明

## 核心判定规则（按优先级）

### 规则1：当天反复（最高优先级）
**触发条件：**
- 上午有异常快照（AM）
- 区域经理标记"已恢复"
- 下午又有异常快照（PM）

**显示：** "多次出问题（当天反复）"

**说明：** 说明门店设备问题没有真正解决，区域经理需要重新处理

---

### 规则2：未及时处理（第二优先级）
**触发条件：**
- 上午有异常快照（AM）
- 下午有异常快照（PM）
- 没有任何处理记录

**显示：** "多次出问题（未及时处理）"

**导出字段：** "未处理日期"列会显示所有未处理的日期（最近10天）

**说明：** 
- 用于追踪区域经理未处理的门店
- 可能是区域经理休假、忙碌等原因
- 复工后可以看到哪些门店需要补处理

---

### 规则3：历史统计（第三优先级）
**触发条件：**
- 5天内出现≥3次异常，或
- 10天内出现≥4次异常

**显示：** "多次出问题（5天3次）" 或 "多次出问题（10天4次）"

**说明：** 
- 排除今天的数据（避免第一天就触发）
- 只统计历史数据
- 用于识别长期反复出问题的门店

---

## 特殊情况处理

### 情况A：标记"未恢复" + 填写预计恢复日期
**行为：** 在预计恢复日期到达之前，该门店不会显示在列表中

**实现：** `should_suppress()` 函数会过滤掉这些门店

**目的：** 减少区域经理重复工作，避免每天都看到同一个门店

**示例：**
- 3月9日上午：门店离线
- 区域经理标记"未恢复"，填写预计恢复日期为3月12日
- 3月9日-3月11日：该门店不会再出现在列表中
- 3月12日：如果还是离线，重新出现在列表中

---

### 情况B：标记"未恢复" + 没填预计恢复日期
**行为：** 继续显示在列表中，但不触发"经常出问题"标记

**目的：** 让区域经理知道这个问题还在，但不算作"经常出问题"

**示例：**
- 门店离线，区域经理标记"未恢复"，但没填日期
- 该门店继续显示在列表中
- 不会被标记为"经常出问题"
- 区域经理可以随时更新处理状态

---

### 情况C：标记"已恢复" + 下午又离线
**行为：** 立即触发"当天反复"（规则1）

**目的：** 快速识别没有真正解决的问题

---

## 数据保留策略

### 快照数据
- 保留最近10天
- 每天最多2次（上午+下午）
- 自动清理超过10天的旧数据

### 处理记录
- 保留最近10天
- 自动清理超过10天的旧记录
- 如果某天没有维护，缺失的日期会被自动忽略

---

## 导出字段说明

| 字段 | 说明 |
|------|------|
| 是否经常出问题 | 是/否 |
| 触发原因 | 多次出问题（当天反复）/ 多次出问题（未及时处理）/ 多次出问题（5天3次）等 |
| 最近5天异常次数 | 包含今天的统计（用于处罚） |
| 最近10天异常次数 | 包含今天的统计（用于处罚） |
| 异常时间点 | 最近5次异常的时间点，格式：03-09下午、03-09上午、03-08下午 |
| 未处理日期 | 所有未处理的日期（最近10天），格式：03-09、03-08 |
| 处理动作 | 已恢复/未恢复/未处理 |
| 未恢复原因 | 区域经理填写的原因 |
| 预计恢复日期 | 区域经理填写的预计恢复日期 |

## 配置参数

所有参数在 `equipment_config.py` 中配置：

```python
# 数据保留天数
SNAPSHOT_RETENTION_DAYS = 10
PROCESSING_RETENTION_DAYS = 10

# 判定规则
ENABLE_SAME_DAY_REPEAT_CHECK = True
ENABLE_UNPROCESSED_CHECK = True
CHRONIC_RULES = [
    {'days': 5, 'threshold': 3},
    {'days': 10, 'threshold': 4},
]

# 预计恢复最长天数
EXPECTED_RECOVERY_MAX_DAYS = 7

# 上午/下午分界点（14点）
AM_PM_BOUNDARY_HOUR = 14
```

修改配置后需要重启服务：
```bash
sudo systemctl restart review-viewer
```

## 使用场景示例

### 场景1：正常处理流程
1. 上午10:30导入数据，发现门店A离线
2. 区域经理联系门店，标记"已恢复"
3. 下午17:00导入数据，门店A恢复正常
4. ✅ 处理完成

### 场景2：当天反复
1. 上午10:30导入数据，发现门店A离线
2. 区域经理联系门店，标记"已恢复"
3. 下午17:00导入数据，门店A还是离线
4. ⚠️ 系统标记"多次出问题（当天反复）"
5. 区域经理需要重新处理

### 场景3：预计恢复
1. 上午10:30导入数据，发现门店A离线
2. 区域经理联系门店，得知需要更换设备，预计3天后恢复
3. 标记"未恢复"，填写预计恢复日期为3天后
4. 接下来3天内，门店A不会再出现在列表中
5. 3天后如果还是离线，重新出现在列表中

### 场景4：区域经理休假
1. 上午10:30导入数据，发现门店A、B、C离线
2. 区域经理休假，没有处理
3. 下午17:00导入数据，门店A、B、C还是离线
4. ⚠️ 系统标记"多次出问题（未及时处理）"
5. 导出时"未处理日期"列显示：03-09
6. 区域经理复工后可以看到哪些门店需要补处理

## 注意事项

1. **只对POS启用历史追踪**：机顶盒只显示简单的离线提示，不做复杂判定

2. **排除今天避免误判**：历史统计（规则3）排除今天的数据，避免第一天就触发

3. **导出包含今天**：导出统计时包含今天的数据，用于准确的处罚统计

4. **去重显示**：异常时间点去重，同一天同一时段只显示一次

5. **容错设计**：如果某天没有维护，缺失的日期会被自动忽略，只保留最近10个自然日的数据
