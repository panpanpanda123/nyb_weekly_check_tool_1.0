## 旧版本（统一处理）

```
┌─────────────────────────────────────────────┐
│ 门店: 测试门店 (ID: STORE001)                │
│ 战区: 华北战区  区域经理: 张三               │
├─────────────────────────────────────────────┤
│ 设备列表:                                    │
│  • [POS] 离线 - 收银机1                      │
│  • [POS] 离线 - 收银机2                      │
│  • [机顶盒] 离线 - 机顶盒1                   │
│  • [机顶盒] 离线 - 机顶盒2                   │
├─────────────────────────────────────────────┤
│ 处理操作: (所有设备统一处理)                 │
│  [✓ 已恢复]  [⚠ 未恢复]                     │

问题:

```

## 处理流程对比

### 旧版本流程

```
用户操作:
  点击 [已恢复] 或 [未恢复]
    ↓
  所有设备统一标记为已处理
    ↓
  无法区分具体哪种设备已恢复
```

### 新版本流程

```
用户操作:
  在 POS 设备组点击 [已恢复]
    ↓
  只标记 POS 设备为已恢复
    ↓
  机顶盒设备仍显示未处理
    ↓
  在机顶盒设备组点击 [未恢复]
    ↓
  输入未恢复原因
    ↓
  只标记机顶盒设备为未恢复
    ↓
  两种设备状态独立记录
```

## 数据库结构对比

### 旧版本

```sql
equipment_processing
├── id (主键)
├── store_id (门店ID)
├── action (处理动作)
├── reason (原因)
└── processed_at (处理时间)

问题: 一个门店只能有一条处理记录
```

### 新版本

```sql
equipment_processing
├── id (主键)
├── store_id (门店ID)
├── equipment_type (设备类型) ← 新增
├── action (处理动作)
├── reason (原因)
└── processed_at (处理时间)

索引: (store_id, equipment_type) ← 复合索引

优势: 一个门店可以有多条处理记录（按设备类型）
```

## API 请求对比

### 旧版本请求

```json
POST /api/equipment/process
{
  "store_id": "STORE001",
  "action": "已恢复",
  "reason": ""
}
```

### 新版本请求

```json
POST /api/equipment/process
{
  "store_id": "STORE001",
  "equipment_type": "POS",  ← 新增
  "action": "已恢复",
  "reason": ""
}

POST /api/equipment/process
{
  "store_id": "STORE001",
  "equipment_type": "机顶盒",  ← 新增
  "action": "未恢复",
  "reason": "网络问题"
}
```

## 导出数据对比

### 旧版本导出

```
门店ID | 门店名称 | 设备类型 | 设备编号 | 处理动作 | 特殊情况理由
STORE001 | 测试店 | POS | POS001 | 已处理 | 
STORE001 | 测试店 | 机顶盒 | STB001 | 已处理 | 

问题: POS 和机顶盒都显示"已处理"，但实际可能只有一种恢复了
```

### 新版本导出

```
门店ID | 门店名称 | 设备类型 | 设备编号 | 处理动作 | 未恢复原因
STORE001 | 测试店 | POS | POS001 | 已恢复 | 
STORE001 | 测试店 | POS | POS002 | 已恢复 | 
STORE001 | 测试店 | 机顶盒 | STB001 | 未恢复 | 网络问题
STORE001 | 测试店 | 机顶盒 | STB002 | 未恢复 | 网络问题

优势: 清楚显示每种设备类型的处理状态
```

## 总结

| 对比项 | 旧版本 | 新版本 |
|--------|--------|--------|
| 处理粒度 | 门店级别 | 设备类型级别 |
| 状态精确度 | 低 | 高 |
| 数据记录 | 1条/门店 | 多条/门店 |
| 用户体验 | 简单但不精确 | 稍复杂但精确 |
| 数据分析 | 困难 | 容易 |
| 问题追踪 | 模糊 | 清晰 |

**结论**: 新版本虽然增加了一点操作复杂度，但大大提高了数据的精确性和可追踪性，更符合实际业务需求。
