## 系统概述

### 功能特点
- ✅ **实时监控**：显示所有离线设备
- ✅ **历史追踪**：识别经常出问题的门店（仅POS）
- ✅ **智能提醒**：支持预计恢复日期，减少重复提示
- ✅ **移动友好**：完整的手机端适配

### 核心逻辑
1. **每天2次检查**：上午10:30、下午17:00
2. **经常出问题判定**：
   - **当天反复**：上午有问题+处理过+下午又有问题 = 立即标记 🔴
   - **5天≥3次**：5天内出现3次或以上异常
   - **10天≥4次**：10天内出现4次或以上异常
3. **数据保留**：最近10天的历史快照和处理记录
4. **自动清理**：超过10天的数据自动删除，节省存储空间

## 数据导入流程

### 📁 文件准备

需要3个Excel文件，放在 `equipment_status/` 目录：

1. **POS设备文件**（必需）
   - 文件名格式：`牛约堡集团_点餐设备表_20260309_1108_xxx.xlsx`
   - 关键列：
     - `组织机构代码`：用于匹配门店ID
     - `设备编号`：设备ID
     - `设备名称`：设备名称
     - `状态`：筛选"离线"

2. **机顶盒文件**（必需）
   - 文件名格式：`202603091108.xlsx`（8位日期+4位时间）
   - 关键列：
     - `设备编码`：用于匹配门店ID
     - `名称`：设备名称
     - `状态`：筛选"离线"

3. **在营门店文件**（必需）
   - 文件名格式：`0306在营门店.xlsx`
   - Sheet名：`营业门店`
   - 关键列：
     - C列：门店ID
     - I列：营业状态（只保留"营业中"）

### 🚀 导入命令

#### 本地测试导入
```bash
# 导入所有设备（POS + 机顶盒）
python import_equipment_data.py

# 只导入POS
python import_equipment_data.py --only-pos

# 只导入机顶盒
python import_equipment_data.py --only-stb
```

#### 服务器导入
```bash

cd /var/www/nyb_weekly_check_tool_1.0

#    数据时间: 2026年3月9日 11:08
# ✅ 找到机顶盒文件: 202603091108.xlsx

# ✅ 设备异常数据导入完成！
```

### 📝 导入逻辑说明

#### 每次导入会做什么？
1. **清空旧设备数据**：删除上次导入的设备异常记录
2. **导入新数据**：只导入离线且在营业中的门店
3. **创建快照**：记录本次检查的异常情况（仅POS）
4. **自动清理**：删除10天前的旧快照和处理记录

#### 为什么要清空旧数据？
- 设备状态是**实时的**，每次导入都是最新状态
- 如果不清空，会累积大量过期数据
- 处理记录和快照都只保留最近10天，自动清理旧数据

#### 如果休息期间没有维护怎么办？
- ✅ **不会报错**：系统会自动忽略缺失的日期
- ✅ **只保留最近10个自然日**：无论中间是否有缺失
- ✅ **自动清理**：每次导入都会清理超过10天的数据

#### 快照的作用
- 快照记录每次检查时哪些门店有异常
- 用于计算"经常出问题"的门店
- 只保留最近10天，自动清理旧数据

### 🗂️ 文件管理

#### 是否需要删除旧文件？
**建议保留最近3天的文件，定期清理旧文件：**

```bash
# 进入数据目录
cd /var/www/nyb_weekly_check_tool_1.0/equipment_status

# 查看文件列表
ls -lh

# 删除7天前的文件（可选）
find . -name "*.xlsx" -mtime +7 -delete

# 或者手动删除旧文件
rm 牛约堡集团_点餐设备表_20260302*.xlsx
rm 202603021*.xlsx
```

#### 文件命名建议
- 保持原始文件名，脚本会自动选择最新的
- 如果同时存在多个文件，脚本会用最新修改时间的文件
- 建议每次上传前删除旧文件，避免混淆

## 区域经理使用指南

### 🔍 查看自己负责的门店

1. 访问：https://weeklycheck.blitzepanda.top/equipment
2. 选择"区域经理"筛选器，输入自己的名字
3. 点击"搜索"

### ✅ 处理已恢复的设备

**场景**：设备已经恢复正常

1. 找到对应门店
2. 点击"✓ 已恢复"按钮
3. 完成！下次导入时，如果设备正常就不会再显示

### ⚠️ 处理未恢复的设备

#### 场景1：短期内会恢复（如：正在维修）

1. 点击"⚠ 未恢复"按钮
2. 填写未恢复原因：`设备维修中`
3. 选择预计恢复日期：`明天` 或 `后天`（最长7天）
4. 点击"提交"

**效果**：在预计恢复日期前，该门店不会再次提示异常

#### 场景2：长期无法恢复（如：门店不配合）

1. 点击"⚠ 未恢复"按钮
2. 填写未恢复原因：`门店不配合，拒绝维修`
3. **不填写**预计恢复日期
4. 点击"提交"

**效果**：每次检查都会继续提示，直到问题解决

### 🔴 经常出问题的门店

**标记说明**：
- 门店名称旁有红色标记：`🔴 当天反复` 或 `🔴 5天3次`
- **当天反复**：上午有问题，处理过了，下午又有问题（最严重）
- **5天3次**：最近5天内出现了3次异常
- **10天4次**：最近10天内出现了4次异常
- 需要重点关注和处理

**如何查看**：
- 卡片模式：红色标记直接显示
- 鼠标悬停可看详细统计：`当天反复` 或 `5天3次 / 10天4次`

## 常见问题

### Q1: 导入后看不到数据？
**A**: 检查以下几点：
1. 确认文件格式正确（列名、Sheet名）
2. 确认门店在whitelist中
3. 确认门店状态是"营业中"
4. 清除浏览器缓存后刷新

### Q2: 经常出问题的门店没有显示？
**A**: 需要满足条件：
1. **当天反复**：上午有异常+处理过+下午又有异常（立即显示）
2. **5天≥3次**：已导入至少5天的数据，5天内≥3次
3. **10天≥4次**：已导入至少10天的数据，10天内≥4次
4. 只对POS设备启用，机顶盒不显示

### Q3: 预计恢复日期不生效？
**A**: 检查：
1. 确认填写了预计恢复日期
2. 确认日期没有过期
3. 如果已过期，会自动恢复提示

### Q4: 处理记录丢失了？
**A**: 只保留最近10天：
- 设备数据每次导入会清空（因为是实时状态）
- 处理记录只保留最近10天，超过10天自动清理
- 如果看不到，可能是已经超过10天被清理了

### Q5: 上午处理了，下午又出现？
**A**: 这是正常的：
- 上午标记"已恢复"
- 下午检查时设备又离线了
- 说明问题没有真正解决
- 建议填写预计恢复日期或标记为"未恢复"

### Q6: 如何修改判定规则？
**A**: 编辑配置文件：
```bash
# 编辑配置
vi /var/www/nyb_weekly_check_tool_1.0/equipment_config.py

# 修改规则（例如改为7天4次）
CHRONIC_RULES = [
    {'days': 7, 'threshold': 4},
    {'days': 14, 'threshold': 6},
]

# 重启服务
sudo systemctl restart review-viewer
```

### Q7: 数据库空间不够？
**A**: 系统会自动清理：
- 只保留最近10天的快照和处理记录
- 可以调整保留天数：
```python
# equipment_config.py
SNAPSHOT_RETENTION_DAYS = 7  # 改为7天
PROCESSING_RETENTION_DAYS = 7  # 改为7天
```

### Q8: 如何清空所有数据重新开始？
**A**: 谨慎操作！
```bash
# 连接数据库
psql -U postgres -d configurable_ops

# 清空设备数据
DELETE FROM equipment_status;

# 清空处理记录
DELETE FROM equipment_processing;

# 清空快照
DELETE FROM equipment_status_snapshot;

# 清空导入日志
DELETE FROM equipment_import_log;
```
