# 设备分类处理功能更新指南

## 更新内容

将设备异常处理功能从"按门店统一处理"改为"按设备类型分别处理"。

### 主要变化

1. **数据库变更**
   - `equipment_processing` 表新增 `equipment_type` 字段
   - 新增复合索引 `idx_processing_store_type`
   - 旧的处理记录将被清空

2. **功能变更**
   - 每个门店的 POS 和机顶盒可以分别处理
   - 每种设备类型独立显示"已恢复/未恢复"按钮
   - 处理记录按设备类型分别保存

3. **界面变更**
   - 设备按类型分组显示（收银系统 POS / 机顶盒）
   - 每个设备组下方有独立的处理按钮
   - 处理状态按设备类型分别显示

使用 WinSCP 上传以下文件到 `/var/www/nyb_weekly_check_tool_1.0/`：

### 4. 重启应用

```bash

# 检查服务状态

```

### 5. 验证功能

1. 访问 https://weeklycheck.blitzepanda.top/equipment
2. 搜索有设备异常的门店
3. 确认：
   - 设备按类型分组显示（收银系统 POS / 机顶盒）

```bash

```

## 注意事项

1. **数据清空**：迁移过程会清空所有旧的处理记录
2. **重新处理**：区域经理需要重新处理设备异常
3. **缓存清理**：浏览器可能需要强制刷新（Ctrl+F5）来加载新的 JS/CSS
4. **测试建议**：先在测试环境验证，确认无误后再部署到生产环境

## 技术细节

### 数据库变更

```sql
-- 添加 equipment_type 字段
ALTER TABLE equipment_processing 
ADD COLUMN equipment_type VARCHAR(50) NOT NULL;

-- 创建复合索引
CREATE INDEX idx_processing_store_type 
ON equipment_processing(store_id, equipment_type);
```

### API 变更

处理设备异常的 API 请求体增加 `equipment_type` 字段：

```json
{
  "store_id": "门店ID",
  "equipment_type": "POS",  // 新增：POS 或 机顶盒
  "action": "已恢复",
  "reason": ""
}
```
